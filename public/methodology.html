Lead Router Methodology

EVALUATION METHOD

Measured, not guessed.

Lead Router uses frozen evidence, labeled ground truth, and a deterministic local scorer. No model judges another model.

01

Every model gets the same test

The benchmark is loaded as data from the allowlisted ASCII Box workflow repository. It contains five frozen, dated cases and eighteen public professional sources. The candidate list, prompt, evidence packet, output contract, and reasoning effort are identical for every accessible model.

Repository code is never executed. A recorded run represents the exact benchmark version shown in its report.

02

Quality is computed from ground truth

The scorer is ordinary JavaScript. Given the same benchmark and the same model output, it returns the same result every time.

Quality 70% role F1 + 20% evidence exactness + 10% company-fit accuracy
Role precision
How many returned people have the expected buying role. It penalizes irrelevant leads.
Role recall
How many expected buyers were found. It penalizes missed leads.
Role F1
The harmonic mean of precision and recall, so returning too many or too few people cannot game the score.
Evidence exactness
A quote counts only when it appears exactly in the supplied source text.
Company-fit accuracy
Whether each company is correctly accepted or rejected against the benchmark label.

03

Bad outputs cannot win cheaply

A completed model is eligible only when its output is structurally valid, its quality is at least 0.75, pricing is known, and measured usage and latency are present.

Recommendation 65% quality + 20% cost efficiency + 15% latency efficiency

Cost and latency are normalized against the best completed value in that run. Ties break by higher quality, then lower cost, then lower latency.

04

Cost and latency are observed

Measured cost
The provider-reported cost for the actual request. A separate estimate uses the dated price snapshot and uncached input, cached input, and output tokens.
Latency
Elapsed time from sending the non-streaming model request until its complete response arrives. It is not time to first token.

05

What is deterministic—and what is not

Deterministic

  • Frozen benchmark cases and labeled ground truth
  • Candidate intersection and output validation
  • Quality formulas, weights, threshold, and tie-breaks
  • Cost reconstruction from recorded token usage

Variable

  • The model response itself
  • Provider routing and service load
  • Token usage, measured cost, and network latency
  • Results across repeated runs
Demo-low confidence. The MVP runs one trial per model on five cases. It proves the evaluation path and one observed tradeoff; it does not provide statistical confidence or a universal model ranking.
